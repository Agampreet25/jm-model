Detailed Theory

The Mathematics Behind the Model

A complete treatment of the J–M model — from historical context to statistical foundations.

Historical Context

In 1972, Z. Jelinski and P.A. Moranda published "Software Reliability Research" — one of the first attempts to apply mathematical reliability theory to software systems. Prior to this, reliability engineering existed only for hardware. The J–M model borrowed from exponential hardware failure models and adapted them to software's unique failure mechanism: latent bugs rather than physical wear.

What is Software Reliability?

Formally, software reliability R(t) is the probability that a software system executes without failure during the time interval [0, t], given a specified operational environment. It is defined as R(t) = P(no failure in [0,t]). Unlike hardware, software does not degrade — it fails only when execution encounters a latent fault, which makes reliability a function of fault density and usage patterns.

Core Assumptions — In Detail

N

Fixed Initial Fault Count (N)

At the start of testing, the software contains exactly N faults. This number is fixed and does not change due to the testing process itself. N is treated as an unknown parameter estimated from observed failure data using Maximum Likelihood Estimation (MLE).

R

Perfect Fault Removal

Each detected fault is immediately and perfectly removed. No new faults are introduced during removal. This means the fault count decreases by exactly 1 after each failure. In reality, fault removal efficiency is typically 0.80–0.95, not 1.0.

E

Equal Fault Hazard Contribution

All remaining faults contribute equally to the hazard rate. Each fault contributes exactly phi to z(t). This homogeneity assumption means a one-line typo and a critical memory corruption bug are treated identically — a significant simplification.

X

Exponential Inter-failure Times

The time X_i between the (i-1)th and ith failures follows an exponential distribution with hazard rate z_i = phi(N-i+1). This means failures are memoryless — the probability of failing in the next instant depends only on the current fault count, not on how long since the last failure.

The Hazard Function — Formal Definition

z(t) = φ · (N − i)
z(t) — Instantaneous hazard rate (failures per unit time)
φ — Per-fault failure rate constant (φ > 0)
N — Initial total fault count (unknown, estimated)
i — Number of faults removed so far (0 ≤ i ≤ N)
N − i — Remaining fault count at the ith interval

The Reliability Function

For the ith inter-failure interval, the reliability (probability of no failure up to time t after the (i-1)th failure) is derived from the hazard rate:

R_i(t) = e^(−φ(N−i+1)·t)
R_i(t) — Probability of surviving to time t in the ith interval
e — Euler's number (approx. 2.718)
φ(N−i+1) — Hazard rate during the ith interval
t — Time elapsed since the last failure

Parameter Estimation via MLE

In practice, N and φ are unknown. They are estimated using Maximum Likelihood Estimation from observed inter-failure times x₁, x₂, ..., xₙ. The log-likelihood function is:

ln L(N,φ) = n·ln(φ) + Σ ln(N−i+1) − φ·Σ(N−i+1)·xᵢ
n — Number of observed failures
xᵢ — ith observed inter-failure time
Setting ∂lnL/∂φ = 0 and ∂lnL/∂N = 0 gives the MLE estimates N̂ and φ̂

The Mean Value Function

The expected number of failures by time t (accumulated across all intervals) gives the mean value function μ(t), which is used to predict total failures over a testing period:

E[failures] = N · (1 − e^(−φt))
As t → ∞, E[failures] → N (all faults eventually found)
The curve is concave — early testing finds faults faster

Interpretation

What the Graph Tells You

The staircase shape is not just visual — each step encodes a precise statistical statement.

01

Monotonically Decreasing Hazard Rate

Each bug fix permanently reduces z(t) by exactly φ. The staircase only ever goes down — never up. This monotonic decrease is a core theorem of the model, derivable directly from the formula.

02

Constant Rate Within Each Interval

Between two consecutive fixes, the hazard rate is perfectly flat. This is the exponential distribution's memoryless property — past survival time does not affect future failure probability.

03

φ Encodes System Fragility

A higher φ means each individual fault has a larger impact. It represents the hazard per fault and is constant across all faults. Complex, tightly-coupled systems have higher φ. Simple, isolated modules have lower φ.

04

Zero Hazard Equals Perfect Reliability

When all N bugs are removed, z(t) = φ × 0 = 0. The model predicts theoretically perfect reliability — an idealization that underscores why the perfect repair assumption is its most criticised simplification.

Reality Check

Where the Model Falls Short

The J–M model is a foundational abstraction. Real-world software rarely behaves this cleanly.

01

Bug Fixes Introduce New Bugs

Studies show up to 15–20% of bug fixes introduce new faults. The assumption of perfect repair (ε=1) rarely holds. More realistic models use imperfect debugging where ε < 1.

02

Non-uniform Fault Severity

The equal-φ assumption is a strong simplification. A critical race condition contributes far more to hazard than a UI typo. Weighted models (like Littlewood-Verrall) address this.

03

Modern Systems Are Non-stationary

CI/CD pipelines ship new code daily, continuously introducing new faults. The fixed-N assumption completely breaks down — N is a moving target in modern development.

04

AI and ML Systems

Neural network failures — distributional shift, adversarial inputs, hallucinations — are not discrete bugs. There is no countable N, no φ, and no fix that restores perfect reliability.

05

Environment and Load Effects

Failure rates depend on workload, concurrency, and environment. The model has no notion of operational profile — a fault may only trigger under 1000 concurrent users.

06

N is Unknown in Practice

N must be estimated from failure data using MLE — but MLE for N is notoriously unstable for small samples. The estimate N-hat has high variance, limiting prediction confidence.

Mathematical Derivation

Step-by-step derivation of the Jelinski–Moranda hazard function

Step 1

Start with the Exponential Failure Model

We model the time between failures as exponentially distributed. For a system with a constant hazard rate λ, the probability density function (PDF) of the time to failure X is:

f(x) = λ · e^(−λx)    for x ≥ 0

And the survival (reliability) function is:

R(x) = P(X > x) = e^(−λx)

Note: The exponential distribution is memoryless — the probability of failure in the next instant does not depend on how long the system has been running. This is the key property that makes the math tractable.

Step 2

Define the Hazard Rate for the ith Interval

Before any bugs are fixed (i = 0), there are N bugs. Each bug independently contributes a failure rate φ. The total hazard rate is the sum of all individual contributions:

z₁ = φ + φ + φ + ... (N times) = N · φ = φ(N − 0)

After fixing the 1st bug (i = 1), only N−1 bugs remain:

z₂ = φ(N − 1)

After fixing the ith bug, N−i bugs remain. Therefore the general formula for the hazard rate in the (i+1)th interval is:

z(i) = φ · (N − i)    for i = 0, 1, 2, ..., N

Note: This is the core J–M formula. It comes directly from the superposition principle: independent Poisson processes (one per bug) can be added together, and their rates sum linearly.

Step 3

Derive the Reliability Function for Interval i

In the ith interval, the hazard rate is z_i = φ(N−i+1). Since we are using an exponential model, the reliability function (probability of no failure up to time t into this interval) is:

R_i(t) = e^(−z_i · t) = e^(−φ(N−i+1)·t)

The PDF of the ith inter-failure time X_i is:

f_i(x) = φ(N−i+1) · e^(−φ(N−i+1)·x)

And the expected (mean) time between the (i−1)th and ith failure is:

E[X_i] = 1 / (φ(N−i+1))

Note: As i increases (more bugs fixed), E[X_i] increases — the system takes longer and longer to fail. This is the reliability improvement made quantitative.

Step 4

Build the Likelihood Function

Given observed inter-failure times x₁, x₂, ..., xₙ, the joint likelihood L(N, φ) is the product of all individual PDFs (since inter-failure times are independent):

L(N, φ) = ∏ᵢ₌₁ⁿ φ(N−i+1) · e^(−φ(N−i+1)·xᵢ)

Taking the natural logarithm (log-likelihood is easier to maximize):

ln L = n·ln(φ) + Σᵢ ln(N−i+1) − φ · Σᵢ (N−i+1)·xᵢ
Step 5

Maximize to Get MLE Estimates

Differentiate ln L with respect to φ and set to zero:

∂(ln L)/∂φ = n/φ − Σᵢ (N−i+1)·xᵢ = 0

Solving for φ̂:

φ̂ = n / Σᵢ (N−i+1)·xᵢ

Differentiating with respect to N and setting to zero gives a transcendental equation for N̂ (solved numerically). Once N̂ is known, plug back in to get φ̂.

Result: The two-parameter model (N, φ) is fully estimated from observed failure data. With these estimates, you can predict future failure rates and determine when to stop testing.

Practice Problems

Numerical problems with complete step-by-step solutions

Problem 1

Computing Hazard Rates After Each Fix

Given: A software system initially contains N = 5 bugs. The per-bug hazard rate constant is φ = 0.02 failures per hour. Failures are observed at times t₁ = 10h, t₂ = 25h, t₃ = 45h after the start of testing.

Find:
(a) The hazard rate z(t) in each inter-failure interval
(b) The expected time to next failure after each fix
(c) The reliability R(t) at t = 5 hours into the 3rd interval
Part (a) — Hazard Rates

Using the J–M formula: z(i) = φ · (N − i)

IntervalBugs Fixed (i)Remaining (N−i)z(i) = φ·(N−i)
1st (before any fix)i = 05 − 0 = 50.02 × 5 = 0.10
2nd (after 1st fix)i = 15 − 1 = 40.02 × 4 = 0.08
3rd (after 2nd fix)i = 25 − 2 = 30.02 × 3 = 0.06
4th (after 3rd fix)i = 35 − 3 = 20.02 × 2 = 0.04

Each fix reduces the hazard rate by φ = 0.02. The staircase drops by the same amount at each step.

Part (b) — Expected Time to Next Failure

For an exponential distribution with rate z_i, the mean inter-failure time is E[X_i] = 1/z_i:

E[X₁] = 1/0.10 = 10.0 hours
E[X₂] = 1/0.08 = 12.5 hours
E[X₃] = 1/0.06 = 16.7 hours
E[X₄] = 1/0.04 = 25.0 hours

The expected time between failures increases with each fix. The system becomes more reliable — it takes longer to fail.

Part (c) — Reliability at t = 5h in the 3rd Interval

In the 3rd interval, z₃ = 0.06. Using the reliability function R_i(t) = e^(−z_i · t):

R₃(5) = e^(−0.06 × 5) = e^(−0.30)
R₃(5) = e^(−0.30) ≈ 0.7408

Answer: R₃(5) ≈ 0.7408 — there is a 74.08% probability that the system runs for at least 5 hours without failure during the 3rd test interval (after 2 bugs have been fixed).

Problem 2

Parameter Estimation from Observed Failure Data

Given: A software module is tested and failures are observed at the following cumulative times (in CPU hours): T₁ = 3, T₂ = 8, T₃ = 18, T₄ = 35 hours. So the inter-failure times are x₁=3, x₂=5, x₃=10, x₄=17 hours.

Assume N = 6 (estimated separately). Find:
(a) The MLE estimate of φ̂
(b) The hazard rate for the 5th interval
(c) The predicted time to the next (5th) failure
Part (a) — MLE Estimate of φ̂

The MLE formula for φ given observed inter-failure times xᵢ and N is:

φ̂ = n / Σᵢ₌₁ⁿ (N − i + 1) · xᵢ

With n = 4 observations and N = 6, compute the denominator term by term:

ixᵢ (hours)N−i+1(N−i+1)·xᵢ
136−1+1 = 66 × 3 = 18
256−2+1 = 55 × 5 = 25
3106−3+1 = 44 × 10 = 40
4176−4+1 = 33 × 17 = 51
Sum134
φ̂ = 4 / 134 ≈ 0.02985 failures/hour/bug

Each of the 6 bugs contributes approximately 0.02985 failures per hour to the total hazard rate.

Part (b) — Hazard Rate for the 5th Interval

After 4 failures (and thus 4 bug fixes), i = 4. Remaining bugs = N − i = 6 − 4 = 2.

z₅ = φ̂ · (N − i) = 0.02985 × (6 − 4)
z₅ = 0.02985 × 2 = 0.05970 failures/hour
Part (c) — Predicted Time to 5th Failure

The expected time to the next failure in the 5th interval is:

E[X₅] = 1 / z₅ = 1 / 0.05970
E[X₅] ≈ 16.75 hours

So we predict the 5th failure will occur approximately 16.75 hours after the 4th failure.

Summary: φ̂ ≈ 0.02985 | z₅ ≈ 0.0597 | E[X₅] ≈ 16.75 hours. Note how the expected inter-failure time has grown from 3h (first failure) to 16.75h — the system is significantly more reliable after 4 bug fixes.
